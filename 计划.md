# RAG 全流程知识库学习计划

## 项目目标

使用 Chroma 向量数据库，从零搭建完整的 RAG（Retrieval-Augmented Generation）知识库，掌握从文档处理到问答生成的完整流程。

## 学习阶段（8步）

### 阶段1：环境与配置

- 使用 **Conda** 创建独立开发环境：`conda create -n rag python=3.11 -y`
- 安装依赖：`chromadb`, `langchain`, `langchain-chroma`, `langchain-community`
- 嵌入模型：优先使用本地模型 `sentence-transformers/all-MiniLM-L6-v2`（节省成本）
- LLM：使用 **DeepSeek-V4 Pro**（API 调用）
- 创建 `.env` 配置文件、`config.py` 统一管理
- 理解 LangChain + Chroma 的集成方式

### 阶段2：文档加载（Loader）

- 使用 `langchain_community.document_loaders` 加载多种格式：
  - PDF（PyPDFLoader）
  - TXT / Markdown
  - CSV
- 了解 LangChain Document 对象的结构（`page_content` + `metadata`）

### 阶段3：文本分割（Splitter）

#### 核心策略：递归字符分割（Recursive Character Splitting）

- 使用 LangChain 的 `RecursiveCharacterTextSplitter`，按优先级依次尝试用不同分隔符切分：
  ```
  分割优先级："\n\n" → "\n" → "。" → " " → ""
  ```
- 即先按段落切，如果段落太大再按句子切，依此类推，最大程度保留语义完整性

#### 关键参数

| 参数 | 说明 | 建议值 |
| --- | --- | --- |
| `chunk_size` | 每个 chunk 的最大字符数 | 500-1000 |
| `chunk_overlap` | 相邻 chunk 之间的重叠字符数 | 50-150 |
| `separators` | 自定义分隔符列表 | 默认即可 |

#### 重叠机制（Overlap）

```
原文: [段落A ...] [段落B ...] [段落C ...]
chunk1: [段落A ...][段落B片段]
chunk2: [段落B ...][段落C片段]
```
- 设置 `chunk_overlap=100` 可避免上下文被硬切断
- 重叠部分帮助跨 chunk 语义连贯

#### 进阶：语义分块（可选）

- 按 `RecursiveCharacterTextSplitter` 做完初分割后
- 合并小于 `chunk_size/2` 的短 chunk，减少碎片
- 对 Markdown 文档可按 `#` 标题层级结构分割（`MarkdownHeaderTextSplitter`）



#### 分块大小的影响

- **过小（<300字符）**：语义不完整，检索精准但上下文缺失
- **过大（>1500字符）**：噪声多，检索精度下降，LLM 处理负担加重
- **调优方法**：先用默认值，根据检索效果逐步调整

### 阶段4：嵌入模型（Embedder）

- 对比本地模型（`sentence-transformers/all-MiniLM-L6-v2`）vs OpenAI（`text-embedding-ada-002`）
- 使用 Chroma + LangChain 集成嵌入
- 理解嵌入向量维度和语义相似度原理

### 阶段5：向量存储（VectorStore）

- 使用 `Chroma` 创建持久化向量库（`persist_directory`）
- `add_documents` 批量写入
- 理解 Chroma 的 Collection 概念
- 实现向量库的增删改查管理

### 阶段6：检索模块（Retriever）

- `similarity_search` 语义检索
- `similarity_search_with_score` 带相关性分数
- MMR（Max Marginal Relevance）多样化检索
- 掌握 `top_k` 参数调优

### 阶段7：生成模块（Generator）

- 拼接检索结果为上下文（context）
- 构建 prompt 模板
- 对接 LLM（**DeepSeek-V4 Pro**）
- 理解 RAG 核心思想：检索增强生成

### 阶段8：完整 RAG 流程整合

- 将前7个模块串联为完整 pipeline
- `main.py` 提供交互式问答 CLI
- 添加简单的查询历史记录
- 测试并优化端到端流程

### 阶段9：单元测试

- 使用 `pytest` 编写单元测试，覆盖每个核心模块
- `test_loader.py`：测试 PDF / TXT / Markdown 加载，验证 Document 对象结构
- `test_splitter.py`：测试分块结果，检查 chunk_size / overlap 边界
- `test_embedder.py`：测试嵌入向量维度、一致性
- `test_vectorstore.py`：测试向量库增删改查、去重
- `test_retriever.py`：测试语义检索结果、MMR 多样化
- `test_generator.py`：测试 prompt 拼接、LLM 调用 mock
- 使用 `pytest.fixture` 复用 Chroma 临时实例，避免测试污染
- 运行 `pytest --cov=src` 生成覆盖率报告

## 扩展方向（基础完成后探索）

### 1. 混合检索
- 原理：BM25（稀疏检索）+ 向量检索（稠密检索）融合
- 使用 `Reciprocal Rank Fusion` 算法合并两个检索结果
- 工具：`rank_bm25` 库 + LangChain 集成

### 2. 重排序（Reranker）
- 目的：向量检索初筛后，用更强大的模型精排
- 常用方案：Cohere Rerank / BGE-Reranker / Jina Reranker
- 在 LangChain 中通过 `ContextualCompressionRetriever` 集成

### 3. 多模态 RAG
- 文档图片 OCR：提取 PDF 中的图片文字（`pytesseract` / `unstructured`）
- 图文检索：使用多模态嵌入模型（如 `CLIP`）实现图片+文字统一检索
- 扩展知识库支持图片问答

### 4. Web RAG
- 使用 `langchain_community.document_loaders` 加载网页内容
- 抓取指定 URL 或站点地图批量构建知识库
- 工具：`AsyncChromiumLoader` / `WebBaseLoader`

### 5. 查询改写（Query Rewriting）
- 目的：用户问题与文档表述不一致时，优化查询
- 方案：HyDE（Hypothetical Document Embeddings）、Query Decomposition
- 使用同一 LLM 生成假设性答案再检索

### 6. 对话记忆（Conversational Memory）
- 多轮对话场景下保留历史上下文
- 实现方式：ConversationBufferMemory / ConversationSummaryMemory
- LangChain 提供的 `ConversationalRetrievalChain` 内置支持

### 7. 生产化部署
- **API 服务**：FastAPI 包装 RAG pipeline，支持 POST 查询
- **异步处理**：文档上传后异步解析入库，Celery 任务队列
- **监控与日志**：结构化日志、调用量统计、延迟监控
- **容器化**：Docker 镜像 + docker-compose 一键部署

### 8. 评估体系
- **RAGAS**：自动化评估 RAG 系统的 Faithfulness、Answer Relevance、Context Relevance
- **手动评估**：构建 Ground Truth问答对，对比生成结果
- **A/B 测试**：对比不同分块策略、嵌入模型、prompt 模板的效果
